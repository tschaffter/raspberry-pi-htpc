Here are the questions that we want to answer:

  • Use Built-in Bluetooth or ASUS USB-BT400 USB Adapter
  • Use USB 2.0 or 3.0 port for ASUS USB-BT400 USB Adapter
  • Use Xbox One S Controller or PS4 Controller
  • Does jstest captures all the Bluetooth packages received from the controller?

Manual preparation of the data

jstest files

These lines are manually removed from the jstest dump files (*_jstest.txt).

  • First 4 lines (general info)
  • Following lines with event type 129 and 130 (list of all buttons)

hcidump files

hcidump generates several lines for each package received. For each package, we only keep the line that includes the keyword “ACL data”. For example,

Helper functions

# Reads Bluetooth data generated by my Bash script.
read_bt_data <- function(filename) {
  read.table(file = filename, sep = '\t', header = TRUE)
}

# Reads jstest files.
# Note: Here we use our own timestamp generated with `ts '%s'`. The timestamp
#       generated internally by `jstest` is the number of ms elapsed since the
#       start of the system.
read_jstest_data <- function(filename) {
  df <- read.table(text = gsub(",", "", readLines(filename)))
  data.frame(
    time = df$V1,
    event_type = df$V4,
    event_number = df$V8,
    event_value = df$V10
  )
}

# Reads hcidump data.
read_hcidump_data <- function(filename) {
  df <- read.table(file = filename, sep = ' ', header = FALSE)
  data.frame(
    time = df$V1
  )
}

We read the data and apply the following transformation to them for each experiment.

  • Set time to 0 when the first controller button is pressed
  • Keep only data between 0 and 300 seconds
  • Start HCI RX and TX bytes at 0 bytes (bt data)
# generate all the combinations of parameters
experiments <- expand.grid(
  hci = names(hcis), 
  device = names(devices), 
  location = names(locations),
  stringsAsFactors = FALSE
)

# create experiment names 
experiment_names <- experiments %>% 
  unite(x, names(experiments), sep = "_", remove = TRUE) %>% 
  pull(x)

# read and transform data
data <- lapply(1:nrow(experiments), function(i) {
  e <- experiments[i,]

  # read raw data
  bt = read_bt_data(file.path(data_dir, paste(
    e$hci, e$device , e$location, "bt.tsv", sep = "_"
  )))
  js = read_jstest_data(file.path(
    data_dir,
    paste(e$hci, e$device , e$location, "jstest.txt", sep = "_")
  ))
  hcidump = read_hcidump_data(file.path(
    data_dir,
    paste(e$hci, e$device , e$location, "hcidump_.txt", sep = "_")
  ))
  
  # set time to 0 when pressed first controller button
  t0 <- js$time[1]
  bt$time <- bt$time - t0
  js$time <- js$time - t0
  hcidump$time <- hcidump$time - t0
  
  # keep bt data collected after pressing the first button until the end of the
  # experiment defined by t_max
  bt <- subset(bt, time >= 0 & time <= t_max)
  js <- subset(js, time >= 0 & time <= t_max)
  hcidump <- subset(hcidump, time >= 0 & time <= t_max, drop = FALSE)
  
  # start HCI RX and TX bytes at 0 bytes
  bt$hci.rx.bytes <- bt$hci.rx.bytes - bt$hci.rx.bytes[1]
  bt$hci.tx.bytes <- bt$hci.tx.bytes - bt$hci.tx.bytes[1]
  
  list(bt = bt, js = js, hcidump = hcidump)
})

# name data
names(data) <- experiment_names

We create a second representation of the data after applying the following transformations:

  • bt data: keep only numeric and variables that may change over time
  • js data: count the number inputs per second
  • hcidump data: count the number of packets received per second

We then pivot and combine these data to make it easier to plot them.

data_ <- do.call(rbind, lapply(1:nrow(experiments), function(i) {
  e <- experiments[i,]
  name <- paste(e$hci, e$device, e$location, sep = "_")
  d <- data[[name]]
  
  # transform, pivot and combine data
  df <- bind_rows(
    d$bt %>%
      select_if(is.numeric) %>%
      select(-c("hci.bluetooth.version", "dev.0.bluetooth.version")) %>%
      pivot_longer(!time, names_to = "name", values_to = "value"),
  
    d$js %>%
      group_by(time) %>%
      summarise(n_inputs = length(time), .groups = 'drop') %>%
      pivot_longer(!time, names_to = "name", values_to = "value"),
    
    d$hcidump %>%
      group_by(time) %>%
      summarise(n_packets = length(time), .groups = 'drop') %>%
      pivot_longer(!time, names_to = "name", values_to = "value")
  )
  
  # append experiment parameters
  df$hci <- e$hci
  df$device <- factor(e$device, levels = c(names(devices)))
  df$location <- e$location
  
  df
}))

HCI data received over time

We show below the amount of bytes received by the built-in Bluetooth of the Raspberry Pi (hdi0, Bluetooth 5.0) and the ASUS Bluetooth adapter (hci1, Bluetooth 4.0). The ASUS adapter is USB 2.0 and connected to a port UBS 2.0 of the Raspberry Pi 4. The ASUS adapter is advertised as supporting up to 3 Mbps (megabits per second).

One of the main differences between the Xbox One S and the PS4 controllers is that the Xbox controller only sends data when buttons are pressed while the PS4 controller constantly sends data. This may come from the built-in accelerometer and gyroscope of the PS4 controller that could be constantly streamed. Note that maintaining the PS4 controller still does not affect the amount of data received (data not shown).

The amount of data received by hci0 and hci1 is almost identical when using the PS4 controller. The amount of data received is 6.4 KB (kilobytes) over 5 minutes. This corresponds to a RX rate of 171.4 Kbps (kilobits per second).

The amount of data received by hci0 and hci1 is very different when using the Xbox controller. As we have observed that the controller only sends data when buttons are pressed, one reason could be that less buttons were pressed when collecting data for hci1 than when doing so for hci0. We don’t have this information but we have the number of buttons pressed / inputs captured by jstest. However, if Bluetooth packets are lost, this would naturally translate into a lower number of inputs reported by jstest. A figure shown below reveals that the number of inputs received from the Xbox controller byhci0 and hci1 is indeed different, though the difference in the RX rate is smaller in the third repetition. Reproducing the entire experiments led to similar results where the RX rate is much lower when using hci1 than hci0, therefore we can not discard the hypothesis that something else contributes to lower the RX rate when using hci1. At the same time, we show with the PS4 controller that we can successfully reproduce a similar number of inputs when using hci0 and hci1 (see below).

The amount of data per packets is larger for the PS4 controller than the Xbox controller. This amount of data is independent of the button pressed. This amount is larger for the PS4 in part because the values of the built-in accelerometer and gyroscope are likely transmitted. We will have a look at the content of these packets later in this document. Here, the RX rate when using the Xbox controller is 28 Kbps for hci0 and 19 Kbps for hci1. Here the rate is computed only over the 3 minutes buttons were pressed.
